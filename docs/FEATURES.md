# Open Review - Current State & Features

## Overview

Open Review is an AI-powered code review tool that generates structured PR reviews with:
- Severity-based findings (Critical, Warning, Info)
- Section summaries generated by AI
- Version tracking and timestamps
- GitHub-native formatting with collapsible sections

---

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    GitHub Actions Workflow                   │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │   Checkout   │ -> │ Open Review  │ -> │    Post      │  │
│  │     Code     │    │    Core      │    │   Comment    │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│                           │                                  │
│                           ▼                                  │
│                   ┌──────────────┐                          │
│                   │  Anthropic   │                          │
│                   │    Claude    │                          │
│                   └──────────────┘                          │
└─────────────────────────────────────────────────────────────┘
```

### Data Flow

1. **GitHub Action** fetches PR code via `actions/checkout`
2. **Open Review Core** generates review using Mastra + Claude
3. **AI generates**: findings + section summaries + verdict
4. **Action formats** output as GitHub-flavored markdown
5. **Action posts/updates** comment on PR

## Current Features

### 1. Structured Review Output

#### Finding Structure
```typescript
interface ReviewFinding {
  id: string;
  type: 'issue' | 'suggestion' | 'praise' | 'question';
  severity: 'critical' | 'warning' | 'info';
  category: string;        // e.g., "security", "performance"
  title: string;           // Short finding title
  description: string;     // Detailed explanation
  file?: string;           // File path
  line?: number;           // Line number
  suggestedFix?: string;   // Code suggestion
}
```

#### Review Result Structure
```typescript
interface ReviewResult {
  summary: string;                    // Overall PR summary
  recommendation: 'approve' | 'request_changes' | 'comment';
  findings: ReviewFinding[];
  tokensUsed: number;
  sectionSummaries?: {                // AI-generated section headers
    mustFix?: string;
    shouldFix?: string;
    questions?: string;
    suggestions?: string;
  };
}
```

### 2. PR Comment Template

#### Sections (in order)
1. **Verdict Line** - LGTM / Changes requested / Hold
2. **Summary** - Brief overview (required when not approved)
3. **Must Fix** - Critical blocking issues (numbered list)
4. **Should Fix** - Non-blocking improvements (bulleted list)
5. **Questions** - Discussion prompts (when type='question')
6. **Suggestions** - Nice-to-have (collapsible by default)
7. **Footer** - Version, timestamp, metadata

#### Section Format
```markdown
🔴 Must fix
[AI-generated summary of critical issues]

1. **Finding Title** — `file:line`
   Finding description...

   **Suggested fix:**
   ```suggestion
   code fix here
   ```

🟡 Should fix
[AI-generated summary of warnings]

- **Finding Title** — `file:line`
  Finding description...

💡 Suggestions (non-blocking)
[AI-generated summary of suggestions]

<details>
<summary>View suggestions</summary>

- **Finding Title** — `file:line`
  Finding description...
</details>
```

### 3. Version Tracking

- Version increments on each re-review (v1, v2, v3...)
- Stored in hidden HTML metadata block
- Displayed in footer: `v4`
- Used to update existing comments instead of creating new ones

### 4. Timestamp & Timezone

- Timestamp shown in footer: `Apr 21, 2026, 1:43 PM`
- Configurable timezone via `timezone` input
- Default: UTC
- Example: `America/New_York` for EST

### 5. Comment Management

- **First run**: Creates new comment
- **Re-runs**: Updates existing comment (finds by `github-actions[bot]` + marker)
- **No duplicates**: Single evolving comment per PR

## Configuration

### GitHub Action Workflow

```yaml
name: Open Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  contents: read
  pull-requests: write

jobs:
  review:
    name: AI Code Review
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: elliottlawson/open-review-action@main
        with:
          provider: anthropic
          model: claude-sonnet-4-20250514
          api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          timezone: America/New_York  # Optional: defaults to UTC
```

### Input Parameters

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `provider` | Yes | - | LLM provider (anthropic, openai, openrouter) |
| `model` | Yes | - | Model name (e.g., claude-sonnet-4-20250514) |
| `api_key` | Yes | - | API key for the provider |
| `conventions` | No | auto-detected | Path to conventions file |
| `ignore` | No | - | Comma-separated glob patterns |
| `timezone` | No | UTC | Timezone for timestamps (IANA format) |

## AI Prompting

### Base Instructions
The AI is instructed to:
1. Explore codebase for context
2. Review changed files with full context
3. Categorize findings by severity
4. Provide specific file paths and line numbers
5. Include suggested fixes when applicable
6. Generate section summaries highlighting key themes

### Section Summary Prompt
```
After listing your findings, generate brief 1-2 sentence summaries for each section:
- mustFix: What's most critical about these blocking issues
- shouldFix: What's most important about these warnings
- questions: What needs discussion
- suggestions: What's suggested for improvement
```

## Known Limitations

1. **Questions Section**: AI rarely generates `type: 'question'` findings. Requires prompt tuning to encourage architectural questions.

2. **Praise**: `type: 'praise'` is filtered out in agent output (agents don't need positive feedback).

3. **File Size**: Large PRs may hit token limits. Review focuses on changed files but reads related files for context.

4. **Line Numbers**: Based on AI's reading; may not match exact diff line numbers in some cases.

## Getting Started

### For Users

1. Add workflow file to `.github/workflows/open-review.yml`
2. Set API key in repository secrets
3. Open a PR - review appears automatically

### For Developers

1. Clone `open-review` repo
2. Run `npm install && npm run build`
3. Test locally: `open-review review --diff main --json`
